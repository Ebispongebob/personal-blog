## 背景

StarLink 是我们这边一个广告归因服务。前阵子有个一次性的活儿，要把旧系统 ClickHouse 里大概 800 万行历史归因数据，一次性搬到 PostgreSQL 里。

导入逻辑不复杂，1 万行一批，从 CH 分页拉取，到 PG 批量 upsert。每一批打上 `import_batch_id` 标记，一来方便按批回滚，二来日常的过期清理任务可以豁免这些历史数据，别误删。

设计上挺合理。

跑起来没一会儿就不对劲了。

## 现象

- 导入越来越慢，跑到 320 万行（不到一半）时已经明显乏力；
- PG 监控「每秒扫描行数」从导入初期的 2600 行/s 一路攀升到 160 万行/s，曲线单调上涨；
- ClickHouse 侧没有任何告警。

直觉判断是「PG 扫描太多导致导入变慢」。排查后发现，这是**两个独立的问题**，只是都随导入进度线性恶化，看起来像一个。

## 根因一：被「豁免删除」的数据，变成了清理任务的死重

系统有个 retention 清理 worker，每 15 秒跑一次，删除 30 天前的过期归因数据。它的查询大意是：

```sql
SELECT ... FROM attribution_results
WHERE import_batch_id IS NULL          -- 导入的历史数据豁免，不删
  AND activate_event_timestamp < 30天前
ORDER BY activate_event_timestamp ASC LIMIT 1000
```

支撑它的索引只建在 `activate_event_timestamp` 上，不含 `import_batch_id`。问题来了：

- 导入的历史数据时间戳都很老，全部落在「30 天前」的范围里，恰好堆在索引扫描的最前端；
- 每一行都要回表检查 `import_batch_id`，然后因为豁免被丢弃；
- 线上实时数据还不满 30 天，压根没有可删的行。

结果就是：每 15 秒，把已导入的全部行从头到尾白扫一遍，一行不删，下个 15 秒再来一遍。已导入 100 万行时每轮扫 100 万，导入到 320 万时每轮扫 320 万——监控曲线的线性攀升就是这么来的。

更糟的是，这不是导入期间的临时痛。不修的话，导入完成后这 800 万豁免行会被永久性地每 15 秒扫一遍。

**修复**：把索引改成 partial index——

```sql
CREATE INDEX CONCURRENTLY ... ON attribution_results (activate_event_timestamp)
WHERE import_batch_id IS NULL;
```

豁免行根本不进索引，清理扫描回到只看实时数据。线上用 `CONCURRENTLY` 在线换，不用停服。

## 根因二：ClickHouse 深分页 OFFSET，导入本身是 O(N²)

导入的拉取用的是经典分页：

```sql
ORDER BY 五个排序列 LIMIT 10000 OFFSET N
```

OFFSET 分页的代价是：数据库必须把前 N 行也「算出来再扔掉」。每批的代价随 offset 线性增长，整个导入的总代价是 O(N²)——offset 到 320 万时，取 1 万行要让 CH 处理几百万行。剩余一半数据的耗时约是已耗时的 5 倍，还可能撞上查询超时导致 job 失败（而失败重跑会从 0 开始，没有断点）。

CH 之所以「没告警」，是因为每条查询都成功了，只是越来越贵。慢查询不等于故障，告警看不到的地方要看趋势指标（如 `system.query_log` 里的 `read_rows` 是否随进度上涨）。

**修复**：改成 keyset 分页（游标分页）——记住上一批最后一行的排序键，下一批用 `WHERE (排序键) > (游标) ORDER BY ... LIMIT 10000`。每批代价恒定，还能借分区裁剪越跑越快。

## 附带发现的隐患：长任务的租约没有心跳

导入 job 用「认领时写 `locked_at`，超过 15 分钟视为僵死可被抢占」来做故障接管，但运行期间从不刷新 `locked_at`。于是任何跑超 15 分钟的健康任务，在多副本部署下都会被别的 worker 当成僵尸抢走，从头并发第二跑。

这次因为单副本单 worker 没有触发，但 800 万行的任务必然超 15 分钟。

**修复**：在每批的进度更新里顺带刷新 `locked_at`，天然心跳。

## 六条可复用的经验

**1.「豁免/排除」型条件必须进索引。**

凡是「扫描时跳过某类行」的周期任务（清理、补偿、对账），被跳过的行如果会大量增长，就是扫描的死重——用 partial index 或把过滤列纳入索引，让「不处理的行」根本不被看见。

**2. LIMIT/OFFSET 只适合浅分页。**

批量搬运数据一律用 keyset/游标分页，代价 O(N) vs O(N²) 的差距在百万行级别是「能跑完」和「跑不完」的差距。

**3. 相关不等于因果。**

「PG 扫描上升」和「导入变慢」都随进度线性恶化，看起来像因果，实际是两个独立 bug 的共同症状。排查时先问：哪个量在随进度线性增长？谁在周期性地碰这个量？用 `pg_stat_statements` 按 rows 排序找真凶，而不是靠猜。

这条我印象最深。人一旦先入为主觉得「就是它」，后面所有证据都会往那个方向解读。两个 bug 凑一起的时候尤其危险——你修了一个，另一个还在，然后你会怀疑自己是不是修错了，其实是压根没修到第二个。

**4. 大批量导入前，把所有周期性后台任务在脑内跑一遍。**

清理、同步、扫表统计——它们面对突然多出几百万行「形状特殊」的数据（老时间戳、特殊标记）时会怎样？这次的坑本质上是导入数据的「老时间戳 + 豁免标记」组合击中了清理任务的扫描路径。

**5. 长任务的租约要续租。**

「超时即抢占」的接管机制必须配心跳，否则健康的长任务就是下一个被「接管」的对象。顺手的做法是让进度上报兼职心跳。

**6. 长任务要有断点。**

从 0 重跑虽然幂等，但在 O(N²) 的世界里「幂等」救不了你的时间。跑到 320 万挂了重跑，又是 320 万行白处理一遍。断点续传不是可选项，是必需品。

## 写在最后

系统设计里有很多「隐式假设」，平时没事，一旦数据「形状特殊」就会集体暴雷。

retention worker 的隐式假设是「需要清理的数据和需要保留的数据量级差不多，索引扫一扫就行」。OFFSET 分页的隐式假设是「翻页不会翻太深」。租约机制的隐式假设是「任务跑得不会太久」。这些假设埋在代码里，平时看不出来，等到边界条件一到，才一个个冒头。

每一个「设计上挺合理」的背后，都埋着一堆当时没人去 check 的隐式假设。合理，是相对于当时的数据形状而言的。数据形状一变，合理就成了坑。

所以第 4 条经验我反复强调，导入前把后台任务过一遍，其实就是在 check 这些隐式假设在新数据形状下还成不成立。这个习惯比任何具体的技术修复都值钱。

这些东西书上不会教，文档里也不会写，都是拿时间换来的。写下来给自己留个底，也给踩同样坑的朋友提个醒。
