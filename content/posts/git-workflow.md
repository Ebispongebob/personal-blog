## 主流 Git 工作流

### Git Flow

Git Flow 是一种经典的分支模型，适合发布周期明确的项目：

```
main ────────────●────────────────────●────────────────
                │                     │
release/1.0  ───┘                     │
                                     │
develop  ────────●────●────●────●────●────●───────────
                      │         │         │
feature/login   ──────●────●────┘         │
feature/payment ───────────────────●────●───────────
```

### Trunk-Based Development

更适合持续部署的团队：

- 所有开发在主干分支进行
- 使用特性开关控制功能发布
- 频繁提交，小步快跑

## 提交规范

使用约定式提交（Conventional Commits）：

```
feat: 新增用户登录功能
fix: 修复密码校验漏洞
docs: 更新 API 文档
refactor: 重构订单模块
```

## 代码审查清单

- [ ] 代码是否符合项目规范
- [ ] 是否有足够的测试覆盖
- [ ] 是否引入了安全风险
- [ ] 性能影响是否可接受
- [ ] 文档是否同步更新

## 总结

选择适合团队的工作流比追求完美更重要。关键是团队成员达成共识，并持续优化协作流程。
