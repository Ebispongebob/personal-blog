本文提出了一种基于 CPU 使用率的 Apache Flink 自动伸缩机制。该机制旨在通过动态调整 Flink 作业的并行度，自动响应负载变化，从而降低运维成本，提高资源利用率和作业稳定性。

## 1 引言

在 Kubernetes 环境中，[HorizontalPodAutoscaler (HPA)](https://kubernetes.io/zh-cn/docs/tasks/run-application/horizontal-pod-autoscale/) 通过自动扩缩工作负载资源（如 Deployment 或 StatefulSet）来满足需求，展现了自动伸缩技术的优势。HPA 的设计理念和控制机制为构建其他计算框架（如 Apache Flink）的自动伸缩方案提供了重要的参考。

Flink 作为流式计算引擎，其作业的资源需求往往随数据流量的波动而变化。传统的手动调整并行度方式不仅耗时费力，而且难以精确匹配实时负载，导致资源浪费或性能瓶颈。因此，实现 Flink 作业的自动伸缩具有重要意义。

## 2 核心算法

借鉴 Kubernetes HPA 的核心算法：

期望副本数 = ⌈ 当前副本数 × (当前指标 / 目标指标) ⌉

我们针对 Flink 场景进行调整，提出基于 CPU 使用率的并行度计算公式：

期望并行度 = ⌈ 当前并行度 × (当前 CPU 使用率 / 目标 CPU 使用率) ⌉

## 3 系统架构

本方案的系统架构包含以下几个核心组件：

1. **指标采集器：**
    - 使用 Prometheus 作为监控系统，通过 PromQL 查询获取 Flink TaskManager 的 `flink_taskmanager_Status_JVM_CPU_Load` 指标。
    - 应用时间窗口（计算过去一段时间内的平均值）对 CPU 使用率进行平滑处理，以消除瞬时峰值的影响。

2. **伸缩决策器：**
    - 扩缩容触发条件：
        - **触发扩容：** `当前 CPU 使用率 > 目标 CPU 使用率 + 容忍波动值`
        - **触发缩容：** `当前 CPU 使用率 < 目标 CPU 使用率 - 容忍波动值`

3. **并行度计算器：**
    - 根据第 2 节提出的公式，计算期望的并行度。

4. **伸缩执行器：**
    - 操作流程：
        1. 从当前 Flink 作业创建 Savepoint。
        2. 停止当前作业。
        3. 以新的并行度从 Savepoint 恢复作业。

## 4 未来研究方向

1. **多指标融合：** 引入更多指标（如内存使用率、背压信号、网络 I/O 等）进行综合决策，提高伸缩的准确性和鲁棒性。

2. **预测性伸缩：** 基于历史负载模式，利用机器学习等技术预测未来的资源需求，提前进行伸缩，进一步优化资源利用率。
