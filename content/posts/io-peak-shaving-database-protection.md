## 背景

前阵子我们在往 PostgreSQL 里导入一批历史数据，数百万行。任务一启动，监控上那条 I/O 使用率的曲线从接近空闲直接弹到 100%，一点缓冲都没有。

第一反应是降并发。worker 从 4 个砍到 2 个，再砍到 1 个。

1 个 worker，I/O 还是满的。

直觉上你会觉得，一个连接而已，数据库不至于扛不住吧。但事实是，只要这条连接能持续不断地提交写请求，它照样能把存储设备的 IOPS 和带宽吃干抹净。数据库不会因为「只有一个连接」就自动放慢写入速度。

后来我们没有改导入总量，只是在每批写入之间加了个 1 秒的暂停。数据库平均 I/O 从接近 100% 降到了 30% 到 40%。总数据量一行没少，导入时间稍微长了一点，但线上业务稳了。

这件事让我想明白一个事，批量任务的真正问题不在总量，在于压力太集中。

这个认知挺反直觉的。大家第一反应都是，总量大那就慢慢导嘛，降并发就好了。但降并发解决的是「多个 worker 叠加放大」的问题，它解决不了「一个 worker 就能吃满」的问题。

## 一次批量写入在数据库里到底发生了什么

要理解这个，得先搞清楚，你业务层看到的是「插入一行数据」，但 PostgreSQL 实际干的活远不止写一张表。

- 表数据页要写
- 主键索引、唯一索引、普通索引都要维护
- WAL 要生成要刷盘
- 事务提交要做 fsync
- 脏页要交给 bgwriter 和 checkpoint 回写
- 如果是 upsert，还有冲突检查
- 如果一条历史记录还要同步写多张状态表，写入量还得翻倍

你看到的一行，在存储层面可能是几张表的写入，加几个索引的更新，加一堆 WAL 操作。

所以一个 worker 持续不断地提交，它产生的写入压力远比你想象的大。这不是数据库的问题，是批量任务本身的特性，它就是会以最高速度往里灌。

## 持续满载挤压后台任务

I/O 满载这件事，最伤的其实不是导入任务本身。导入任务巴不得 I/O 拉满，它跑得越快越好。

真正受伤的是数据库的那些后台任务。

PostgreSQL 有一堆后台任务在持续运行，checkpoint 在回写脏页，bgwriter 在提前刷脏，autovacuum 在清理旧版本数据，WAL 在刷盘和归档，主从复制在同步。这些任务和你的导入任务共享同一套 CPU、内存和存储。

导入任务把 I/O 长期占满之后，checkpoint 没法平滑回写脏页了，只能让脏页越积越多。积到一定程度，checkpoint 开始集中追赶，这时候数据库延迟会突然升高，查询超时、连接堆积、复制延迟，一个接一个来。

这些后台任务不会报错，不会告警，它们只是默默地变慢。等你发现的时候，往往已经是一连串的问题了。

## 在线业务也需要 I/O 余量

还有一个容易被忽略的，是在线业务。

生产数据库不是只跑你的导入任务，它还在承载实时业务。用户事件写入、实时归因、状态更新、API 查询、管理后台查询，这些请求的数据量可能不大，但对延迟极其敏感。

导入任务关心吞吐量，在线业务关心响应时间。两者共用一个数据库的时候，导入任务把 I/O 吃满，在线请求只能跟它在磁盘队列里抢位置。

结果就是 SQL 延迟突然上升，接口变慢，连接池开始堆积，上游服务开始超时重试，重试又进一步放大数据库压力。一个连锁反应就这么起来了。

## 削峰：在批次之间留出空档

我们的处理方式很简单，在每个导入批次完成之后，加一个固定暂停时间。

配置类似：

```yaml
config:
  importBatchSize: 2500
  importWorkers: 1
  importBatchInterval: "1s"
```

导入流程从连续写入变成了，写一批，commit，暂停 1 秒，再写下一批。

总数据量没变，每批的写入动作也没消失，但压力被摊到了更长的时间窗口里。

暂停那 1 秒里，数据库可以喘口气，消化 I/O 队列，刷 WAL，平滑回写脏页，跑 checkpoint 和 bgwriter，处理 autovacuum，响应在线业务请求。

最终结果就是导入时间增加了一些，但数据库平均 I/O 降到了 30% 到 40%，在线业务有了明显的资源余量。

## 为什么平均 I/O 会下降

假设每批写入耗时 W 秒，批次间暂停 P 秒，那导入任务的活跃时间占比大概是 `W / (W + P)`。

比如每批写入 0.6 秒，暂停 1 秒，活跃占比就是 `0.6 / 1.6 ≈ 37.5%`，跟实际观察到的 30% 到 40% 基本对得上。

写入那一下的瞬时 I/O 还是可能很高，但暂停阶段给数据库留了恢复窗口，监控周期内的平均使用率和排队时间都降下来了。

## Batch、Worker、Interval 分别控制什么

批量任务有三个参数特别容易搞混。

**Batch Size** 管的是单次事务处理多少数据。批次太大，单事务持续时间长，WAL 突增，内存压力高，失败重跑成本也高。但光减小 batch size 是没用的，批次变小了下一批还是立刻跟上，数据库照样满载，而且事务提交次数还变多了。

**Worker 数量** 管的是同时跑几个导入任务。减 worker 能避免并发放大，但一个不限速的 worker 照样能打满数据库。所以低并发不等于低负载。

**Batch Interval** 管的是每批完成后等多久，也就是持续写入速率。它不减少总数据量，但直接改变压力在时间轴上的分布。三个里面，它是削峰最有效、也最容易调的参数。

## 不要追求满载

很多人对批量任务有个误区，觉得把数据库跑满才是「资源利用充分」。

对离线计算任务，跑满 CPU 或磁盘可能确实没问题，反正那个集群就是干这个的。但对同时承载在线业务的生产数据库，长期 100% I/O 绝对不是一个健康目标。

更合理的思路是，给在线业务保留稳定的资源余量，给 checkpoint、bgwriter、autovacuum 这些后台任务留出执行空间，接受批量任务完成时间适当延长。如果数据库还跑着实时业务，平均 I/O 控制在 50% 到 70% 以下通常更稳妥。具体阈值看实例规格、业务峰值和延迟要求。

## 写在最后

这件事让我想到一个挺普遍的道理。

总量不变，分布变了，结果就完全不同。

水库泄洪的时候，同样的水量，一次性倾泻下去就是洪灾，分成几天慢慢放就是灌溉。交通管制也是，同样的车流量，全挤在一个时段就是堵死，错峰出行就是顺畅。

数据库 I/O 也是一样的道理。你改变不了要写多少数据，但你能改变这些写入在时间轴上怎么分布。

而那些被你「削峰」保护下来的东西，checkpoint 的平滑回写，bgwriter 的提前刷脏，autovacuum 的清理，在线业务的响应延迟，它们不会像导入任务那样在监控面板上大声宣告自己的存在。它们是默默运转的底层设施，平时你不注意，一旦被挤压到失灵，连锁反应才会让你意识到它们有多重要。

批量任务可以慢一点，但不能通过长期吃满数据库资源，把成本转嫁给在线业务和后台维护任务。
