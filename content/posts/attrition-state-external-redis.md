---

先说个让人头大的事。

我们的归因管线跑在 Flink 上，状态全存在 RocksDB 里。这事儿一开始没毛病，数据量小的时候，Flink 自带的状态后端又快又省事，本地读写，连网络都不用走。

但业务跑着跑着，不对劲了。

---

第一个问题，状态在无限膨胀。

归因这个业务有个特殊的地方，它要支持「再归因」。意思是同一个用户，第一次被归到某个渠道之后，过段时间如果又点了新的广告，可能要重新归因。这就导致用户的归因状态不能跟着窗口过期就清掉，得一直留着。

你想想看，用户量是持续增长的，每个用户的状态又不能清，那 Flink 内部维护的这个状态体积就只能往上走。今天 100GB，下个月 150GB，半年后可能就奔着 TB 去了。

状态一大，作业的资源消耗跟着飙，稳定性也往下掉。Flink 的 checkpoint 会变慢，作业重启恢复会变慢，稍微有点抖动就可能触发反压，一反压就堆积，一堆连锁反应。

---

第二个问题更隐蔽，但实际干活的时候更扎心，状态不可观测。

Flink 的内部状态对外部系统是封闭的，就是个黑盒。你想查一下「这个用户当前的归因状态是什么」「这条事件为什么没被归上」，对不起，没法实时查。

线上出了问题，排查全靠猜。要么去翻日志，日志里不一定打了你要的字段。要么去对数据，对半天发现是状态里某个过期数据没清干净导致的。这种黑盒排查，效率低得让人想砸键盘。

---

这两个问题摆在一起，核心矛盾其实就一句话，状态和计算耦合得太死了。

Flink 既是计算引擎，又是状态存储，两件事揉在一块。计算这边想要轻快点，被状态拖着。状态这边想要可观测，被 Flink 的封闭性挡着。你想动任何一个，都得连着另一个一起改。

所以我们做的决定是，存算分离。

把 Flink 从「有状态计算」改成「轻状态」甚至「无状态」计算，把归因逻辑里必须持久化的那些数据，全部下沉到 Redis 集群里。Flink 只管算，Redis 只管存，各司其职。

---

改造前后的架构对比大概是这样。

以前是 Flink Source 进来，KeyBy 分流，然后 ProcessFunction 带着 RocksDB 状态做归因逻辑，最后 Sink 出去。状态都在 TaskManager 本地的 RocksDB 里，读写是快，但就是你一个人的快，外面谁都看不见。

改完之后变成 Flink Source 进来，KeyBy 分流，然后算子里改成读写 Redis 加上简化的处理逻辑，最后 Sink 出去。状态不在本地了，每次要用就去 Redis 取，写完再塞回去。多了一次网络 IO，但换来的是状态完全透明，谁都能查，谁都能改。

这个取舍是值得的。归因这个场景不是那种超高 QPS 的热路径，多一次 Redis 往返，毫秒级，业务完全扛得住。但你换来的是，任何一个用户的归因状态，你拿个 redis-cli 连上去就能看，线上排查从「猜」变成「查」，这个效率提升不是一个量级。

---

思路定了，落地分两个阶段。

第一阶段是改 Flink 作业本身。

先搭基础设施，基于 Jedis 客户端封装一套 Flink 读写 Redis 的通用工具类。这个不难，Jedis 本身 API 就很简洁，主要是把连接池管理、重试、超时这些工程细节处理好，别让 Redis 的抖动直接打到 Flink 的算子上。

然后是业务逻辑重构。原来 Flink State 里的 MapState 和 ValueState，要映射到 Redis 的 Hash 和 String 上。MapState 对应 Hash 很自然，field 就是原来的 key，value 还是原来的 value。ValueState 对应 String，整个序列化成 JSON 塞进去就行。

这一步真正费劲的不是映射本身，是把归因的核心算法，点击归因、再归因这些，用 Redis 命令重新实现一遍。原来在 ProcessFunction 里直接操作 State 的代码，全得改成 Redis 交互。逻辑没变，但写法完全不一样，而且要考虑 Redis 操作失败、网络超时这些原来不存在的异常路径。

最后把原来的 Stateful ProcessFunction 整个移掉，换成带 Redis 交互的新算子。

---

第二阶段是历史数据迁移，这个比改代码更操心。

新作业上线不能从零开始，不然历史归因状态全丢了，老用户的再归因全乱套。所以得先把存量数据灌进 Redis，这叫冷启动。

数据源我们用的 ClickHouse，因为历史事件数据本来就在那存着。范围是清洗后的转化事件全量数据，就是那种「用户被广告带来转化」的记录。写个批处理脚本，把这些历史归因状态按新的 Redis 结构预写进去，构建一个初始状态基准。

这一步有个坑要注意，历史数据的时间对齐。你得算清楚，Redis 里灌进去的状态对应到哪个时间点，然后新作业从那个时间点之后开始消费流式数据。对不齐的话，要么重复归因，要么漏归因，都是事故。

灌完之后，新作业就可以以「无状态」模式轻量启动了。因为状态都在 Redis 里，Flink 这边不需要恢复任何 checkpoint，直接接上流式数据开干就行。这其实也是存算分离的一个额外好处，作业重启变快了，不用等半天 checkpoint 恢复。

---

写到这里差不多讲完了，但我想多说一句关于取舍的话。

存算分离这个方向，这两年在数据基建里几乎成了政治正确，谁都在讲。但落到具体业务上，它不是银弹。

你把状态从本地 RocksDB 搬到远程 Redis，读写延迟必然上升，从微秒级到毫秒级，这是实打实的代价。如果你的业务是那种每秒几十万 QPS、对延迟极其敏感的热路径，这个改动可能得不偿失，你得上更复杂的方案，比如本地缓存加 Redis 兜底。

但归因不是这种场景。它的吞吐没那么夸张，延迟容忍度也够，多一次 Redis 往返完全 OK。而它换来的东西，状态可观测、作业轻量化、重启快、扩容简单，每一条都是实打实的收益。

架构改造这事，从来不是「哪个方案更先进」，而是「哪个方案跟你业务的痛点对得上」。我们的痛点是状态膨胀和黑盒排查，Redis 外置正好打在这两个点上，所以它对我们就是对的。

把黑盒打开这一步，值。
